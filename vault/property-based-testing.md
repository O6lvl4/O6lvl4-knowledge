「こうなるはず」というルールを決め、無数のサンプルで自動的に検証するテスト手法。

## 何ができる？／なぜ重要？

学校の抜き打ちテストを想像してください。普段の小テストは「この問題と、この問題」と内容が決まっていますが、抜き打ちテストは「範囲のどこから出るか分からない」状態で、どんな問題でも答えられるかを試します。プロパティベーステストはこの考えを極端に推し進めて、「ありとあらゆる入力でルールが守られているか」を機械が自動で大量に試す方法です。

たとえば「逆順にした文字列をもう一度逆順にしたら元に戻るはず」というルールがあるとします。普通のテストでは "abc" → "cba" → "abc" のように、人間が考えたいくつかの例で確かめます。プロパティベーステストでは、機械が空文字、絵文字、長文、変な記号、と何百通りも自動生成して、すべてでルールが守られるか試します。さらに失敗を見つけたら、最小の反例まで自動で縮めてくれます。「思いつかなかったケース」で起きるバグを、人手では到達できないスケールで掘り出せるのが強みです。

## 仕組み

```mermaid
flowchart TB
    ルール[性質: f(g(x)) == x] --> 生成[ランダム入力を大量生成]
    生成 --> 実行[実装を実行]
    実行 --> 判定{性質を満たす?}
    判定 -->|満たす| 次へ[次の入力へ]
    判定 -->|破れた| 縮小[反例を最小化]
    縮小 --> 報告[最小反例を報告]
```

性質（プロパティ）を一つ書くと、生成器がランダムに入力を作り、実装に流して結果を確かめます。性質が破れたら、その入力をできるだけ小さくシュリンクして、人間が読める形で報告します。

## 用語

- **プロパティ**: 「常に成り立つはずの性質」。テストの主題。
- **ジェネレータ**: ランダム入力を作る道具。
- **シュリンク**: 失敗例をできるだけ小さく縮める処理。
- **反例**: 性質を壊す具体的な入力。
- **QuickCheck**: プロパティベーステストの元祖ライブラリ（Haskell）。
- **fast-check / hypothesis**: JS や Python で広く使われる実装。
- **不変条件**: 状態が変わっても保たれる性質。
- **往復則**: 「行って戻れば元に戻る」型の性質。
