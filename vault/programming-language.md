人間がコンピュータに対する命令を記述するための人工言語。このノートは vault 内の言語・言語理論ノートへの**入口（ハブ）**。

構文は主に[[type-2-grammar|文脈自由文法]]で定義され、型チェックや名前解決など意味的な処理は[[type-1-grammar|文脈依存]]以上の問題になる（→ [[chomsky-hierarchy|チョムスキー階層]]）。

## 言語を分類する3つの軸

言語は単一の分類に収まらず、独立した複数の軸の**組み合わせ**で位置づけられる。

### 1. 実行方式（どう動かすか）

| 方式 | 仕組み | 例 |
|---|---|---|
| コンパイル（ネイティブ） | 機械語に変換してから実行 | [[rust\|Rust]]、C、Go |
| コンパイル（バイトコード/VM） | 中間表現に変換し VM が実行 | Java、C#、[[grain\|Grain]]（WASM） |
| トランスパイル | 別の高級言語へ変換 | [[almide\|Almide]]（→Rust/TS/JS/WASM）、TypeScript |
| インタプリタ | ソースを直接解釈実行 | [[ruby\|Ruby]]、Python |
| JIT | 実行時に機械語へ動的コンパイル | V8（JS）、PyPy、JVM HotSpot |

実体はグラデーション。多くの「インタプリタ言語」も内部でバイトコードへコンパイルしてから VM で回す（Ruby/CRuby、Python/CPython）。コンパイル／インタプリタは言語そのものの性質ではなく**実装の選択**である点に注意。

### 2. 型システム（誤りをいつ・どれだけ捕まえるか）

2つの独立した軸がある。よくある混同（「動的＝弱い」）は誤りで、両者は直交する。

| | 静的型付け（実行前に検査） | 動的型付け（実行時に検査） |
|---|---|---|
| **強い型付け**（暗黙変換を許さない） | [[rust\|Rust]]、Haskell、OCaml | [[ruby\|Ruby]]、Python |
| **弱い型付け**（暗黙変換が多い） | C（キャストで型を欺ける） | JavaScript（`"1"+1="11"`） |

- **静的 / 動的** = 型検査の**タイミング**（コンパイル時 / 実行時）。
- **強い / 弱い** = 型の**境界の堅さ**（暗黙の型変換・型の踏み破りをどれだけ許すか）。
- 型推論（[[hindley-milner\|Hindley–Milner]]）があれば静的型付けでも注釈をほぼ書かずに済む。型理論の体系的入門は [[tapl\|TAPL]]。

### 3. パラダイム（計算をどうモデル化するか）

| パラダイム | 計算の捉え方 | 代表 |
|---|---|---|
| 命令型 / 手続き型 | 状態の逐次的な書き換え | C、[[nadesiko\|なでしこ]] |
| オブジェクト指向 | 状態と振る舞いを持つオブジェクトの相互作用 | [[ruby\|Ruby]]、Java |
| [[functional-programming\|関数型]] | 式を値に評価する。関数が第一級 | Haskell、OCaml、[[clojure\|Clojure]] |
| 論理型 | 事実と規則からの導出 | Prolog |
| [[stack-oriented-programming\|スタック指向]] | オペランドスタックの操作 | [[forth\|Forth]]、[[mind\|Mind]] |
| 並行・[[actor-model\|アクター]] | メッセージを送り合う独立プロセス | Erlang / Elixir |

ほとんどの実用言語は**マルチパラダイム**（例：[[rust\|Rust]] は命令型＋関数型＋一部 OO 的）。パラダイムは言語が**第一級で何を扱うか**——状態・オブジェクト・関数・述語・スタック——の違いとして読める。

## 概念軸（言語を語るときの語彙）

言語の「動き」を理解するには3層を分けて考える。

- **構文（syntax）** — 何が文法的に正しい文字列か。主に [[type-2-grammar|文脈自由文法]]で定義し、[[parser-combinator|パーサコンビネータ]]や [[pushdown-automaton|PDA]] ベースのパーサで認識する。
- **意味論（semantics）** — プログラムが何を**意味する**か。操作的意味論（[[abstract-machine|抽象機械]]の状態遷移として定義）／表示的意味論（数学的対象への写像）／公理的意味論（事前・事後条件）に大別。
- **評価戦略（evaluation strategy）** — 式をいつ評価するか。正格（先に評価）vs 遅延、値呼び vs 名前呼び。理論的な核は [[lambda-calculus|λ計算]]の[[reduction|簡約]]順序。

型システムは構文と意味論の橋渡し——[[curry-howard|カリー＝ハワード対応]]では「型＝命題、プログラム＝証明」となり、言語理論と論理学が一致する。

## このナレッジベースにある言語

- [[almide|Almide]] — LLM コード生成に最適化。Rust/WASM にコンパイル
- [[rust|Rust]] — 所有権システムで GC なしメモリ安全。システムプログラミング
- [[ruby|Ruby]] — プログラマの幸福を一次価値にした動的型付け言語
- [[lisp|LISP]] — コード = データ。1958年生まれ
- [[grain|Grain]] — WebAssembly ネイティブの関数型言語。LLVM 不使用
- [[forth|Forth]] — [[stack-oriented-programming|スタック指向]]の元祖。1970年
- [[mind|Mind]] — 日本語 × スタック指向
- [[nadesiko|なでしこ]] — 日本語 × 手続き型

## 押さえどころ（カード化候補）

- **3つの分類軸** → 実行方式（コンパイル/VM/トランスパイル/インタプリタ/JIT）・型システム・パラダイムは**独立**。一言語は各軸での座標の組み合わせ。
- **コンパイル/インタプリタは実装の選択** → 言語の性質ではない。Ruby も Python も内部はバイトコードへコンパイルして VM で回す。
- **静的/動的 と 強い/弱い は直交** → 前者は型検査のタイミング、後者は暗黙変換の許容度。「動的＝弱い」は誤り（Python は動的かつ強い）。
- **概念3層** → 構文（[[type-2-grammar|CFG]] で定義）／意味論（操作的・表示的・公理的）／評価戦略（正格 vs 遅延、[[lambda-calculus|λ計算]]の[[reduction|簡約]]順序）。
- **マルチパラダイムが主流** → パラダイムは「第一級で何を扱うか」（状態・オブジェクト・関数・述語・スタック）の違い。
- **理論への接続** → 構文は [[chomsky-hierarchy|チョムスキー階層]]、型は [[tapl|TAPL]] / [[curry-howard|カリー＝ハワード]]、関数型は [[lambda-calculus|λ計算]]に根を持つ。

## 関連

### 言語理論・形式の基盤

- [[chomsky-hierarchy|チョムスキー階層]] — 構文の表現力の形式的分類。言語の文法は主に Type 2
- [[lambda-calculus|λ計算]] — 関数型言語と評価戦略の理論的基盤
- [[reduction|簡約]] — 式を値に書き換える計算のステップ（評価戦略の核）
- [[abstract-machine|抽象機械]] — 操作的意味論を状態遷移で与える（SECD/CEK 等）

### 型・論理

- [[tapl|TAPL]] — 型とプログラミング言語の標準教科書
- [[hindley-milner|Hindley–Milner]] — 注釈なしで型を推論する関数型言語の型推論
- [[curry-howard|カリー＝ハワード対応]] — 型＝命題、プログラム＝証明。言語理論と論理の一致

### パラダイム

- [[functional-programming|関数型言語]] / [[pure-functional-programming|純粋関数型言語]] — 式と値・参照透過性
- [[stack-oriented-programming|スタック指向]] — オペランドスタックで計算する系統（Forth/Mind）
- [[actor-model|アクターモデル]] — メッセージ駆動の並行計算モデル

### パーサ・構文処理

- [[parser-combinator|パーサコンビネータ]] — 小さなパーサを合成して文法を認識する
- [[pushdown-automaton|プッシュダウン・オートマトン]] — Type 2（CFG）を受理する機械。構文解析の理論的土台
