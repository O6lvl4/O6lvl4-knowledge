ノート間の `[[wikilink]]` 接続を統制するための内規。グラフが「便宜的な集積」ではなく「事実関係の地図」になるよう、以下を厳守する。

## リンク採用の判定基準

ある `[[X]]` から `[[Y]]` へのリンクは、以下のいずれかが成立するときのみ採用する：

1. **直接の依存・利用**: Y の実装が X を要素として使う／X の機能を呼び出す
2. **直接の構造関係**: Y が X の部分集合・上位概念・実装例（明示的に README/コードに現れる）
3. **明示された対比**: X が Y を「対」として説明している（例: famulus → famulus2）

以下は **採用しない**：

- 「広義の」「～的」「通じる」「近い発想で」「関連プロジェクト」など、抽象的な類似性のみ
- 「キャプチャ系だから codec 関連」のような分野連想
- 同じ言語で書かれているだけ（Go の repo 同士 等）
- 同じトピックを扱う「ような気がする」関係

## 説明文のルール

各 `- [[Y]] — 説明` の説明部分は、READMEまたはコードから検証できる **具体的な事実** を1行で記す。

採用例:
- `[[almide-wasm-bindgen]] — Almide コードを WebAssembly 経由で JS から呼ぶためのバインディング生成器`
- `[[gv]] — Go ツールチェインのバージョン管理 CLI（uv 級速度を目指す Rust 実装）`

不採用例（剪定対象）:
- `[[X]] — ～関連プロジェクト`
- `[[X]] — ～を活用したツール`
- `[[X]] — ～的な存在`
- `[[X]] — ～に通じる発想`

## セクションの分量

概念ノートの「vault 内での使われ方」は **3〜8 件** を目安。ない場合は省略してよい。
リポジトリノートの「関連」は **3〜6 件** を目安。

数を増やすために弱いリンクを足すことを禁止。**少なく正確に** が原則。

## 既知の頻出ミスパターン

過去のミスから以下を特に警戒する：

| 誤った理解 | 正しい理解 |
|---|---|
| `gv` が Graphviz 関連 | Go version manager (`[[gv]]`) |
| `ccgrid` が AST/grid 表示 | Claude Agent SDK のチーム管理ツール |
| `almide-lander` がランディングページ | Almide の cross-language package exporter |
| `almide-js` が JavaScript ランタイム | Almide の JS/TS バインディング進捗ログ |
| `gulp-coach` が LLM コーチング | 完全ローカル（LLM 不使用）の水分摂取ガイド |
| `famulus2` が RAG | tree-sitter による Code Graph（広義の RAG ではない、別概念） |
| `premaid` がストリーミング | Mermaid を Puppeteer でレンダリングする CLI |

## 違反の発見と是正

リンクが上記基準を満たさない場合：
1. 削除する（推奨）
2. 別のセクション（例「思想的な対比」）に移し、説明を具体化する

## 適用範囲

このルールはすべてのノートに適用される。今後ノートを追加・更新する際も、このルールに従って `[[wikilink]]` を選定すること。
