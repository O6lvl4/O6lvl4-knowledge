## refine は二相: 深掘り → 研磨

refine は **深掘り(enrich)+ 研磨(refine)** の複合操作。順序が大事:

1. **深掘り (enrich)** — ノートが薄い箇所を特定し、**検証済みの実体**(数値・固有名・式・機構・トレードオフ)で埋める。
2. **研磨 (refine)** — 既存 + 追加分をまとめて、冗長除去・再構成・表現研磨・密度向上・一貫性で磨く。

量(深掘り)→ 密度(研磨)の順。**薄ければ行数は増えてよい**が、増える1行は必ず実体を運ぶこと。padding で行数を稼ぐのは禁止。逆に元から厚いノートなら深掘りはスキップして研磨だけでよい。

## 深掘りの観点 (enrich)

### 何を足すか

- 薄い主張に**具体**を: 数値・ベンチマーク・固有名・式・年代・出典
- 概念の**なぜ**が欠けていれば足す(設計判断の理由、トレードオフ)
- 重要な関連概念・限界・落とし穴・対抗手段が抜けていれば補う
- 既存 vault ノートとの接続(`[[wikilink]]`)で文脈を厚くする

### 足す前に検証する

- 外部の事実・数値・固有名は、書く前に出典(論文 / 一次情報 / コード)を当たって確認する。**曖昧な記憶で書かない**
- 確証が取れないものは「未確認」と明示するか、書かない
- 出典 URL は Links セクションに残す

## 研磨の観点 (refine)

以下の観点で、既存 + 深掘りした内容をまとめて磨く。

### 1. 冗長さの除去

- 同じことを別の言い方で繰り返している箇所を統合
- 読者が既に知っている前提を長々と説明している箇所を圧縮
- 「つまり」「要するに」の後に本質があるなら、前の説明が冗長な証拠 — 前を削って後だけ残す
- 修飾語の過剰: 「非常に重要な」→「重要な」or そもそも修飾なしで伝わるなら削る

### 2. 構造の再編成

- セクションの順序は「読者が知りたい順」になっているか
  - 一般原則: 「何であるか」→「なぜ必要か」→「どう動くか」→「比較・トレードオフ」→「押さえどころ」→「関連」
  - この順序に合っていないなら並べ替える
- 1つのセクションに異質な話題が混在していたら分割
- 逆に、細かすぎるセクション分割は統合 (2-3行のセクションが連続していたら)
- 見出しの粒度を揃える (h2 と h3 が混在して階層が不明瞭なら整理)

### 3. 表現の研磨

- 曖昧な記述を具体化: 「パフォーマンスが良い」→ 何と比較して、どの指標で？
- 受動態を能動態に: 「〜が使われている」→「X が〜を使う」
- ぼやけた主語を明確に: 「一般的に〜」→ 誰が？どの文脈で？
- 技術的に不正確な表現を修正 (ただし確信がある場合のみ)

### 4. 知識密度の向上

- 散文で書かれた比較はテーブルに変換できないか検討
- 箇条書きの羅列に構造（グルーピング、順序）を与える
- 長い説明文を Mermaid フローチャートに置換できないか検討 (Mermaid 内に `[[wikilink]]` を使わないこと)
- ただし可読性とのトレードオフ。テーブル化・図化が逆に読みにくくなるならやらない

### 5. 一貫性の確保

- vault 全体のスタイルと統一されているか:
  - 冒頭は「何であるか」の 1-2 文要約
  - コード例にはコメントで文脈を示す
  - 関連セクションは `[[wikilink]]` + ダッシュ + 関連理由の 1 行説明
  - 押さえどころは「トピック → 太字の要約」形式
- 文体のぶれ (敬体/常体の混在、漢字/ひらがなの揺れ) を統一

## アンチパターン (やるな)

- **padding**: 同義反復・自明な一般論・感想で行数を稼ぐ。深掘りは実体を足す作業であって、水増しではない
- **未検証の追加**: 出典を当たらず曖昧な記憶で数値・固有名を書く
- **スコープ膨張**: ノートの主題から逸れた話題を足して焦点をぼかす
- **意味の変更**: 表現を変えるとき、元の意図を変えてはいけない
- **過度な圧縮**: 密度を上げるのは良いが、理解に必要な文脈まで削ると価値が下がる
- **スタイルだけの変更**: 読者体験が変わらない書き換えは無駄。変更には目的が必要

## 手順

1. 対象ノートを読み、**薄い箇所(深掘り対象)と磨くべき箇所(研磨対象)の両方**を洗い出す
2. **深掘り**: 不足を検証済みの実体で埋める。外部の主張は出典を当たってから書く
3. **研磨**: 既存 + 追加分を上記 5 観点で磨く
4. `updated_at` を現在時刻に更新する — 時刻まで含めた ISO 8601 (例: `2026-05-29T15:03:37+09:00`)。recent ビューは同じ日の中でこの時刻順に並べるため、日付だけだと日内の順序が壊れる
5. 深掘りで足した実体と、研磨で締めた点を分けて報告する
